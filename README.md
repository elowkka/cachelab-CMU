##CMU-cacheLab##
##-该项目是卡内基梅隆大学的一个课程项目-##

目标：实现一个缓存模拟器，能够替换缓存中的数据，并且统计命中hit、未命中miss、驱逐eviction 的次数。
并且要优化 32×32、64×64、61×67 矩阵的转置函数，使未命中次数尽可能低。

将其拆解为两部分，PartA与PartB

##PartA##
PartA 要写一个 csim.c 文件，实现一个高速缓存模拟器，模拟缓存的替换并统计命中、未命中与驱逐的次数，并且能够根据不同的s、E、b进行初始配置。首先定义结构体Cache、Set、Line，Cache中包含s、E、b的参数以及指向Set的指针，Set中包含指向Line 的指针，这里的Set就对应上了缓存中的组，而Line就对应上了缓存中的行，即一个具体的缓存数据，其中包含有效位v，标签位tag和LRU的时间戳time。

定义完结构体后，我们先写一个初始化缓存空间的函数Init_cache(int s,int E,int b)，能够根据输入的s、E、b申请对应大小的空间用于存放缓存，并对缓存空间中各参数的值初始化为零。

![image](https://github.com/elowkka/cachelab-CMU/blob/main/png/%E7%BC%93%E5%AD%98%E7%A9%BA%E9%97%B4%E7%A4%BA%E6%84%8F%E5%9B%BE.png)

初始化缓存空间后，我们着手开始实现对缓存的读取和替换，即实现access_cache(Cache cache,int op_tag,int op_set,int E)函数。该函数首先要判断对于传进来的set与tag是否命中，若命中，则命中次数+1；若未命中，则miss次数+1，并且需要判断是冷不命中还是冲突不命中，若为冲突不命中，eviction 次数+1。不命中时，还需要判断是否有空位来存入缓存，因此写了一个判断是否有空位的函数is_empty(Cache cache,int op_set,int E)，若有空位，则直接存入；若无空位，则要按实验指导中的 LRU 方法进行判断，将上次被读取时间最久的位置进行替换。

这里就需要用到我们之前在Line 里定义的时间戳time 了，每次访问缓存，未被访问到的位置的time就要+1，若被访问到了，则时间戳time清零。因此time最大的就是上次读取时间最久远的位置，我们就替换这个位置。因此还需要写一个寻找最大时间戳的函数find_max_time(Cache cache,int op_set,int E)来帮我们找到要被替换的位置。 

此外，还有 print_help()，free_cache()函数来打印帮助信息和手动清理缓存空间。在主函数中，用到了内置函数getopt来读取输入的缓存模拟参数和tracefile文件，将输入的指令解析为s、E、b、t、v等参数。之后进行tracefile文件的读取，将读到的数据进行解析，分块为指令类型op_type，地址块address和大小size，再将地址块分块为tag、set，根据不同的op_type，执行不同次数的缓存访问。最后手动清零缓存空间，打印我们统计的命中、未命中和驱逐的次数。

##PartB##

先来分析给出的示例，示例函数实现了转置的功能，但在32×32测试中的未命中数就达到了1183次。给出的缓存空间s=5，E=1，b=5，对应的S=25=32，B=25=32，有着32 组，每组内有32个字节的block偏移量。由于int占据4字节，因此一组可以放入8 个int。

![image](https://github.com/elowkka/cachelab-CMU/blob/main/png/32_32%E7%9F%A9%E9%98%B5%E4%B8%AD%E7%BC%93%E5%AD%98%E7%9A%84%E5%88%86%E7%BB%84%E6%83%85%E5%86%B5%20.png)


以 32×32 矩阵为例，A 矩阵按行读取，这是好的，读取 A[0][0]，会将其后的A[0][1]……A[0][7]都加载进缓存的一组中，除了 8 个 int 的首个数据未命中，后面的7个int都能命中，理论上的不命中次数为32×32÷8=128次。B矩阵是按列写入，行与行之间有32个int，这就导致每次加载进缓存组的8个int用不上，按列的下一个数据总是未命中。又因为32组的缓存空间意味着最多能保存8行的数据而不产生冲突不命中，当B遍历了第一列后，早就将前几行的缓存驱逐了，因此开始的第二列写入依然都是不命中，所以B矩阵始终是不命中的，理论上不命中次数为32×32=1024次。 

显然，直接进行按行按列的矩阵转置是不可行的，会产生大量的不命中。

由于实验文件中提到，可以对不同矩阵大小进行单独优化，这里先对32×32矩阵进行优化。

(1)32×32 矩阵 
上文提到，缓存空间有32组，对于32×32矩阵来说，最多可以容下8×32的数据，更大的数据就会发生覆写造成冲突不命中了。因此考虑用8×8分块来处理，每次只处理A与B的一个8×8块。

![image](https://github.com/elowkka/cachelab-CMU/blob/main/png/A_B%E7%9F%A9%E9%98%B5%E5%AF%B9%E5%BA%94%E7%9A%84%E9%9D%9E%E5%AF%B9%E8%A7%92%E5%9D%97%E5%8F%8A%E5%90%84%E7%BB%84%E7%9A%84%E4%BD%8D%E7%BD%AE.png)

先以非对角块为例，如图所示，A矩阵的8行会用到组0、组4、组8……组28，而B矩阵会用到组1、组5……组29，二者之间不会产生冲突。理论上它们都只有行/列的第一个数据会产生不命中，后面七个int都能命中，只会产生128×2=256次不命中。这里进行了测试，发现最终的miss数量为343，高于理论值，一个很大的原因是对角块的存在。我们之前的分析中，A、B 矩阵的对应非对角并不会产生冲突，但是在对角块上，由于A、B矩阵之间的地址差为32的整数倍，它们对应的set_index在对角块上就相同了，又由于E=1，一个组只有一行，所以A的缓存会被B驱逐，因此会多产生更多不命中，因此需要特殊处理对角块。

题目允许我们使用最多12个int类型变量，除了循环用到的4个之外，我们刚好可以用8个变量来先存储A的行数据，这样B驱逐A的缓存也不会导致不命中。经测试，miss数量减小为287，达到了满分标准。但是仍有可以优化的地方，我们对A的读取已经做了很好的优化，还可以对B的写入进行优化。对角块上，由于A是行读取，B是列写入，列写入需要8行缓存，二者总会有交错的时候，交错时A的先读取就会让B多一次不命中。因此可以考虑先将B也按行写入，再进行单独的转置，并且由于按行写入时已经将B矩阵的8×8数据加载进缓存，理论上单独转置不会产生更多的不命中。运行最终的代码，miss次数为259次，已经很接近理论的次数256了。 

(2)64×64 矩阵

该矩阵一行有64个int，因此缓存空间最多只能同时容纳4行的数据而不产生冲突。这里就想到了要用4×4分块进行处理，但是4×4分块的不命中率至少都有25%，理论的不命中次数就无法达到满分的要求，更不用提还有对角块等冲突的影响。并且4×4分块对缓存的利用率不高，缓存的一组能容纳8个int，4×4分块却只利用了其中4 个。因此考虑还是用8×8分块，但是为了避免后四行对前四行缓存的覆写，将8×8分块分为4个4×4分块，依次处理各分块，获得更大的缓存利用率。 

![image](https://github.com/elowkka/cachelab-CMU/blob/main/png/8_8%E5%8C%BA%E5%9D%97%E5%88%86%E6%B3%95.png)

首先将A的第1、2块转置复制给B，此时B的第1块就处理好了，但B的第2块存放的本该放在第3块，所以接着将B的第2块转移到第3块，此时还有4个变量
可用，就将A的第3块转置复制到B的第2块。最后再将A的第4块转置到B的第4块，就完成了一个8×8分块。上面的操作中，由于8个变量的使用以及4×4分块不会造成很大的冲突，不命中次数能够大大减少。经测试，最终代码的不命中次数为1179次，达到了满分的要求。

(3)61×67 矩阵

该矩阵并不是2的整数幂，但是满分的要求不高，不命中次数只需要小于2000即可，因此简单使用了8×8分块，并对剩余行列直接转置处理。最终代码的不命中次数为1889次，达到了满分的要求。
